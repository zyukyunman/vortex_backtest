# 17 · 回测对 vortex_data 的数据需求描述

> 交付对象：**vortex_data 服务开发者**。
> 作者视角：vortex_backtest（会话式回测引擎）。
> 目的：明确"会话式回测在每个模拟时刻需要哪些数据、以什么方式取、如何防未来函数"，
> 据此让 data 服务把数据**按访问模式落盘 + 提供按时间闸门的取数能力**。
> 取代 [07-vortex_data-qlib-export-requirements](07-vortex_data-qlib-export-requirements.md)（qlib 已删，见 [14](14-engine-decision-revisit.md)）。

---

## 0. 一句话需求

回测引擎按**模拟时钟**逐步推进；每一步策略会问 data 要"**截至当前模拟时刻**、某些股票、某些字段、最近 N 条"的数据，
有时还要"**全市场某一刻**排序/筛选后的结果"。data 服务需要保证两点：

1. **point-in-time 正确**：任何一次取数都不能返回"当时还不知道的信息"（防未来函数）。
2. **按访问模式分区**：不同数据集的查询形状不同，落盘布局要匹配，否则全市场扫描会慢到不可用。

> 原则上"策略什么数据都可能要"，所以**全集都要可取**；本文不是裁剪数据，而是给每个数据集
> 标注**正确的取数口径**和**合适的存储布局**。

---

## 1. 硬要求一：point-in-time 可见性（防未来函数）

这是回测的生命线。每个数据集必须有一个明确的**"可见时间戳"**——即"这条记录在哪一刻才算
'已经发生 / 已经公开'"。回测只允许取 `可见时间戳 ≤ 模拟当前时刻` 的数据。

最容易踩的坑是**财报和事件**：它们有"报告期/数据期"和"公告期"两个时间，**必须用公告期当可见时间戳**。

| 数据类别 | ❌ 错误口径（会泄露未来） | ✅ 正确可见时间戳 |
|---|---|---|
| 财报（资产负债表/现金流/利润表/财务指标/快报/预告） | 报告期末 `end_date`（如 Q1=03-31） | **公告日 `ann_date`** |
| 分红 events | 报告期 `end_date` | **除权日 `ex_date`**（registry 已是此优先级）；"是否分红"的知晓用 `ann_date` |
| 日线行情/估值/资金/因子/筹码/涨跌停 | —— | **交易日 `trade_date`**（当日盘后可见；盘中只可见到当前 bar） |
| 分钟行情 | —— | **分钟戳 `datetime`**（只可见 ≤ 当前分钟的 bar） |
| 龙虎榜/涨停板/热度榜 | —— | **交易日 `trade_date`**（盘后榜单，次日才可用于决策） |
| 成分股/权重/行业分类 | 用"今天的成分股"回溯过去 | **生效区间**（`in_date`/`out_date`）→ 取"当时的成分股" |

> 落地建议：data 在每个数据集里**显式保留这个可见时间戳列**，并在元数据里声明哪一列是
> "可见性时间戳"。回测取数时统一按它做 `≤ sim_time` 过滤。registry 里已有
> `date_field_priority`（如 events=`[ex_date, …]`），把它**升格为每个集都必须有的"可见性字段"声明**即可。

## 2. 硬要求二：按访问模式分区（性能）

你点出的关键区分——**有些数据只按时间维就够，有些必须个股×时间**——正是分区设计的核心。
按"回测会怎么查"把数据集分成五类，每类给推荐布局。

### 类 A · 纯时间维（无个股，量小）
全市场只有一条时间序列。直接**按时间维存单表/单文件**即可，不需要按股票分区。

- 成员：`calendar`(交易日历)、`shibor`/`shibor_lpr`(利率)、`cn_cpi`/`cn_gdp`/`cn_m`/`cn_pmi`/`cn_ppi`(宏观)、`us_tycr`(美债收益率)、`moneyflow_mkt_dc`(大盘资金)、`moneyflow_hsgt`(北向汇总)。
- 频率：日 / 月 / 季（宏观）。
- 量级：KB～低 MB。
- 取数形状：`给我截至 sim_time 的最近 N 条`。
- 布局：**按 date 单序列**。宏观季度/月度数据注意 `ann_date` 可见性（GDP 等是滞后公布的）。

### 类 B · 少量主体 × 时间（指数/行业）
主体数量很少（几十～几百个指数/板块），按时间存即可，可选按主体二级分区。

- 成员：`index_daily`(指数日线)、`index_global`(全球指数)、`sw_daily`(申万行业日线)、`dc_index`、`ths_index`、`index_basic`(指数列表)。
- 取数形状：`某指数最近 N 日` / `所有指数某日`。
- 布局：**按 date 分区**；主体少，单文件即可全主体扫描。

### 类 C · 个股 × 日（主力，量中～大）★
这是回测日频/盘后决策的主战场，**两种查询都频繁**：
①"某股票深历史"(算指标) ②"全市场某一日"(选股/排序)。

- 成员：`bars`/`pro_bar`(日线 OHLCV)、`adj_factor`(复权因子)、`stk_limit`(涨跌停价)、`suspend_d`(停复牌)、`valuation`(估值/市值)、`moneyflow`(个股资金)、`stk_factor_pro`(技术因子, 247M)、`cyq_chips`/`cyq_perf`(筹码)、`stk_nineturn`、`st`/`stock_st`/`namechange`(风险/曾用名)、`weekly`/`monthly`(周月线)。
- 取数形状：`某股最近 N 日` **和** `全市场某日全部`。
- 布局：**建议双布局**——按 `symbol` 分区（服务"单股深历史"，开 1 文件，快）+ 按 `date` 分区镜像（服务"全市场某日"，开 1 文件，快）。
  - 若不愿双份：至少保证**按 date 分区**，因为"全市场选股"是最差查询，单股历史可以靠下推聚合补偿。

### 类 D · 个股 × 分钟（最大，5.1G 且在涨）★★
`stk_mins` 一个集就占 5.1G。回测两种用法都要：①"某股一段时间的分钟序列"(分钟择时) ②"全市场某一分钟"(分钟级全市场扫描)。

- 成员：`stk_mins`（外加未来若有 `stk_auction_o/c` 集合竞价）。
- 现状：**按 symbol 分区**（`year/universe/symbol=XXX/data.parquet`，~7500 标的）。
  - 利："某股深历史" = 开 1 文件，快（正好是分钟择时场景）。
  - 弊："某一分钟全市场" = 要开 ~7500 个文件各读 1 行，**很慢**（正好是全市场扫描场景最差布局）。
- 布局：**必须双布局**。在现有 by-symbol 之外，再生成一份 **by-date 分区**（`date=YYYYMMDD/`，内含当日全部股票全部分钟），"某日/某分钟全市场" = 读 1 文件。
- 这是全文最值得投入的一项：扫描性能完全由它决定。

### 类 E · 事件/稀疏 + 财报/周期（个股 × 事件日 / 报告期）
稀疏（只在事件发生日有行）或周期（每季一行）。量不大，但**可见性时间戳是重点**（见第 1 节）。

- 事件稀疏：`events`(分红, 可见=`ex_date`)、`top_list`/`top_inst`(龙虎榜)、`limit_list_d`(涨跌停明细)、`limit_step`(连板梯队)、`kpl_list`、`dc_hot`/`ths_hot`(热榜)、`disclosure_date`(披露计划)。
- 财报周期：`balancesheet`、`cashflow`、`income`、`fina_indicator`、`forecast`、`express`（全部可见=`ann_date`）。
- 取数形状：`某股截至 sim_time 最新一期财报` / `某日全市场龙虎榜`。
- 布局：按 date（事件）/ 按 symbol+report_period（财报）皆可，量小不挑布局；**关键是带 `ann_date` 并按它过滤**。

### 类 F · 参考维表（缓变，几乎不随时间动）
策略要用来定义股票池、行业归类、基础信息。

- 成员：`instruments`(标的列表)、`stock_company`、`fund_basic`、`index_member_all`/`index_weight`(指数成分/权重)、`index_classify`、`ths_member`/`dc_member`(板块成分)、`authoritative`(权威表)。
- 取数形状：`sim_time 时点的成分股/分类`。
- 布局：维表，整表小；但**成分/权重要带生效区间**（`in_date`/`out_date`），回测取"当时的成分"，否则用了未来才纳入的股票=选股泄露。

## 3. 对 data 服务的取数能力需求（API 形态）

回测会话每步会发起取数。希望 data 提供一个**统一的、带时间闸门的取数接口**，覆盖下列形状。
（传输用 REST/JSON 还是 Arrow Flight 待定；先约定语义，性能层后议。）

**输入**（一次取数请求应能表达）：
- `dataset`：数据集名（上表任一）。
- `symbols`：股票列表 / `"universe"`(当前会话股池) / `"all"`(全市场)。
- `fields`：所需字段子集（少传少算）。
- `as_of` = sim_time：**服务据此强制 `可见时间戳 ≤ as_of`**（防未来函数的总闸门，由服务保证，不信任调用方）。
- `window`：`count=N`(最近 N 条) 或 `[start, end]` 区间（end 仍受 as_of 截断）。
- `level`：`daily` / `1min`（仅行情类需要）。
- `op`（可选，**算子下推**）：在服务端先算后回，如 `rank/topN/filter/agg`，**只回结果不回原始行**——这是全市场扫描不传海量数据的关键。

**输出**：
- 列式批量（建议 Arrow，回退 JSON）；
- 附带每条的可见时间戳，便于回测核对；
- 全市场聚合类只回算完的结果集。

**三类典型查询都要支持**（对应回测三场景）：
1. **股池切片**：`某几只 × 最近 N 条`（类 C/D 的 by-symbol 布局，快）。
2. **全市场快照**：`all × 某一刻 × 部分字段`（类 C/D 的 by-date 布局，快）。
3. **全市场算子下推**：`all × 某一刻 × rank/topN`（服务端算，回 top 结果，省传输）。

> 注：这是给 data 的**取数语义**需求；回测对策略侧暴露的会话接口（sessions/data/advance/close）在
> design/18（待写）。两者解耦：data 只管"按时间闸门高效给数据"，不感知回测会话。

## 4. 量级预判（为容量/性能定调）

以现状（约 80 交易日样本）实测换算：

- 全市场**每分钟** ≈ 7500 行 ≈ 0.3–0.5 MB；全市场**单日分钟** ≈ 180 万行 ≈ ~50 MB（压缩）；**一年分钟** ≈ ~12 GB 起。
- 每步真正回传很小：股池 200 只某一刻 ≈ 几十 KB；股池 20 根历史 ≈ ~200 KB。
- 只有"全市场 × 深历史 × 每分钟"才重 → 用**算子下推**化解（回结果不回行）。

**结论**：取数本身不是瓶颈，**不必为账户做物理物化表**（存储翻倍 + 时钟每步重写，写放大严重）。
真正的性能杠杆是**类 D 的 by-date 双布局**。

---

## 5. 不要做 / 避免

- ❌ **不要给回测开数据库直连/裸读**。行业主流（QuantConnect "Time Frontier"、Zipline Pipeline）都是
  "时间前沿"——只交付 ≤ 当前模拟时刻的数据，由服务做闸门，不让策略自由读全库（agent 策略尤其控不住未来函数）。
- ❌ **不要按账户物化数据副本**（per-account 表）：分钟级一天数百次推进 = 数百次追加/重写，得不偿失。
- ❌ **不要用行存库（Postgres）扛分钟列式数据**；要直读得上列式（DuckDB/ClickHouse），但仍优先走"取数 API + 共享 parquet"。
- ⚠️ 若将来确需"高级用户写 SQL"，给**受控 SQL 端点**：服务自动注入 `可见时间戳 ≤ as_of` 守卫、跑在受限视图上——"直读"的安全版，不暴露原始库、不复制数据。

## 6. 优先级建议（给 data 排期）

| 优先级 | 事项 | 理由 |
|---|---|---|
| P0 | 每个数据集声明**可见时间戳字段**（尤其财报=`ann_date`、分红=`ex_date`、成分=生效区间） | 不做就有未来函数，回测结果不可信 |
| P0 | 类 C/D 提供**按 date 分区**布局（行情/因子/估值/资金 + 分钟） | 全市场选股/扫描的最差查询，决定可用性 |
| P1 | 统一**取数 API**：`dataset+symbols+fields+as_of+window+level`，服务端强制时间闸门 | 回测会话的取数底座 |
| P1 | 取数 API 支持 **`op` 算子下推**（rank/topN/filter/agg） | 全市场分钟扫描不传海量行 |
| P2 | 类 D 保留 by-symbol 的同时补 **by-date 双布局** | 兼顾单股深历史 + 全市场快照 |
| P2 | 列式传输（Arrow Flight）替代 JSON | 大批量取数提速，可后置 |

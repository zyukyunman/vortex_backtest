# 回测引擎能力清单与引擎选型结论

## 0. 先回答你的问题

> "你是否有信心梳理清楚一个回测框架要实现的东西？如果不行就用 qlib。"

**有信心——下面 §2 就是完整梳理。** 但我要诚实地把结论说在前面：**正因为能把它梳理清楚，我反而把推荐从"薄自研核心"改成了"用 Qlib 做引擎 + 一层很薄的 A 股规则层"。** 理由不是"我做不出来"，而是这张能力图清楚地显示——**最容易出错、最值钱的那几块（复权口径、公司行为、防未来函数、撮合/账户），恰好是 Qlib 已经做好并经过实战的部分**；而我们手写引擎里已经查出 2 个 qfq 正确性 bug（见 `design/01` C1/C3），这本身就是"自研这类东西很容易埋雷"的实证。

再叠加你的两个新输入，天平更明确地偏向 Qlib：

1. **你愿意把 `vortex_data` 也改成 Qlib** → 这消除了我原 ADR-1 反对 Qlib 的主要理由（数据格式两套、转换重复）。而且 `vortex_data/design/04` **本就把"Qlib view export"列为 v1.1**、仓里已有 `qlib_view.py` 桩——统一到 Qlib 是把既定计划做完，不是绕路。
2. **两仓同栈 = 少很多跨栈排查**（你的原话），这点我原来权重给低了，实际很重要。

所以本文件的结论：**推荐 Qlib + 薄规则层，但用 1 天的 spike 先验证唯一真正的不确定点（见 §4），通过就锁定、并据此把 ADR-1 改为 Accepted。** 这不是走流程，是因为"用 Qlib 做*订单回放*"是 Qlib 的次要用法（它的主轴是"信号→组合"），值得花半天到一天确认它不会在 A 股约束上跟我们较劲。

---

## 1. 一个前提：把"轮子"分清楚

"要不要自研引擎"问错了问题。真正的问题是：一个 A 股分钟级**订单回放/账户模拟**引擎，必须实现的能力里，**哪些是任何框架都要你自己写的 A 股规则（不可消除）**，哪些是**成熟框架已经做好、自研只会重新埋雷的通用基础设施**。§2 就按这个口径逐项标注"谁提供"。

---

## 2. 回测引擎能力全景

标注口径——**谁提供**：`Qlib自带`（开箱）｜`Qlib可配`（其参数/扩展点）｜`规则层(薄·必写)`（任何方案都要自己写的 A 股规则）｜`成熟库`（empyrical 等）｜`待spike`（需验证）。**易错度**：自研时埋雷的概率。

### A. 时间与事件模型
| 能力 | 谁提供 | 易错度 |
|---|---|---|
| 交易日历（节假日/周末/半日市） | Qlib自带（A 股日历）/ 也可用 vortex_data 日历 | 低 |
| 分钟时序、连续竞价时段对齐 | Qlib自带（分钟数据） | 低 |
| 集合竞价（9:15–9:25 开盘价、14:57–15:00 收盘价）bar 处理 | 待spike（Qlib 分钟数据如何含/标竞价） | 中 |
| 多标的共享时钟、跨标的日界对齐 | Qlib自带 | 中 |
| **防未来函数**（成交/估值不能用未来信息） | Qlib自带（executor 按时点推进） | **高**（自研最易错） |

### B. 数据与价格口径
| 能力 | 谁提供 | 易错度 |
|---|---|---|
| raw / qfq / hfq 切换 | Qlib自带（factor 复权） | 中 |
| **"哪个价用于哪一步"**（信号/成交/估值/限价/tick 对齐口径分离） | Qlib有既定约定 | **高**（现有代码就栽在这：C1/C3） |
| 复权因子应用、基准固定（不随窗口漂移） | Qlib自带 | 中 |
| 缺 bar / 停牌当日价处理 | Qlib可配 | 中 |
| PIT / 数据时点（对齐 vortex_data PIT） | Qlib自带 | 中 |

### C. 持有期公司行为（最容易被低估的一块）
| 能力 | 谁提供 | 易错度 |
|---|---|---|
| **现金分红入账**（券商真账本：分红打进现金） | ⚠ Qlib 走"复权价总收益"口径，不单记现金流；要真账本需规则层补 | **高**（现有代码**直接跳过**） |
| 送转（股数变化） | Qlib 通过 factor 隐含；真股数账本待确认 | 高 |
| 配股 / 增发 | 多数框架不自动处理 | 高（需定范围） |
| 退市 / 新股上市 | Qlib 数据层 | 中 |

### D. 订单模型
| 能力 | 谁提供 | 易错度 |
|---|---|---|
| 市价 / 限价单 | Qlib Order | 低 |
| 价格类型 open/close/vwap/twap | Qlib executors（Close/Vwap/TWAP） | 低 |
| 订单生命周期（部分成交/拒单/撤单） | Qlib Exchange/Order | 中 |
| **外部订单逐分钟回放**（`order_file` 灌入） | Qlib自带（order-execution 回测路径）——**正是本服务的核心契合点** | 中 |

### E. 撮合与成交
| 能力 | 谁提供 | 易错度 |
|---|---|---|
| 成交价确定、滑点 | Qlib Exchange（`deal_price`） | 中 |
| 量能参与上限、部分成交 | Qlib Exchange（`volume_threshold`） | 中 |
| 防未来成交时点（用本 bar 还是下一 bar） | Qlib executor | 中 |

### F. A 股微结构规则（不可消除的规则层）
| 能力 | 谁提供 | 易错度 |
|---|---|---|
| **T+1**（当日买不可卖、精确解锁到下一交易日） | ⚠ `规则层/扩展`（Qlib **默认不强制** T+1）+ 待spike | **高** |
| 涨跌停（分板 ±10/5/20/30、ST、新股首日无限制、**以数据为准**；涨停不可买/跌停不可卖） | Qlib可配（`limit_threshold`）+ 分板差异化需`规则层`扩展 | 高 |
| 手数（主板 100 / 科创 200 起+1 递增 / 北交所） | `规则层`（Qlib `trade_unit` 是单值，分板差异需扩展） | 中 |
| tick 0.01 对齐（对 **raw 价**，不是 qfq 价） | `规则层` | 低（现有 bug 点） |
| 停牌 / ST / 退市可交易性 | Qlib可配（tradable） | 中 |

### G. 费用
| 能力 | 谁提供 | 易错度 |
|---|---|---|
| 佣金 + 最低 5 元 | Qlib（`open_cost/close_cost/min_cost`） | 低 |
| 印花税（仅卖出） | Qlib可配 / `规则层` | 低 |
| 过户费 | `规则层`/配置 | 低 |
| 按券商/按回测可配 | `规则层` | 低 |

### H. 账户与持仓
| 能力 | 谁提供 | 易错度 |
|---|---|---|
| 现金 / 持仓 / **可卖量 vs 总量** | Qlib Account/Position（可卖量要配合 T+1） | 中 |
| 成本基准（移动平均/FIFO） | Qlib Position | 中 |
| 挂单冻结资金/股份 | Qlib | 中 |
| 多账户 / 多策略独立账本 | `服务层编排`（自研，本就有） | 中 |

### I. 估值与净值
| 能力 | 谁提供 | 易错度 |
|---|---|---|
| 逐 bar 市值、日结 | Qlib | 低 |
| 分钟 / 日净值曲线 | Qlib report + `成熟库` | 低 |
| 停牌按最后价估值 | Qlib可配 | 中 |

### J. 绩效与风险指标
| 能力 | 谁提供 | 易错度 |
|---|---|---|
| 收益/年化/回撤/Sharpe/Sortino/Calmar/换手/胜率/基准相对 | `成熟库`（empyrical-reloaded / quantstats）或 Qlib report | 低 |

### K. 边界与偏差
| 能力 | 谁提供 | 易错度 |
|---|---|---|
| 未来函数防护（框架级保证） | Qlib | **高**（自研最易错） |
| 幸存者偏差 / 退市样本 | `数据层`（vortex_data） | 中 |
| PIT / 财报重述 | `数据层` | 中 |

## 3. 逐域结论与总体判断

把上表压缩成一句话：**绿区（Qlib 自带/可配）几乎覆盖了全部"高易错"子域**——防未来函数、复权口径、撮合、账户、费用、估值、指标；**红区（规则层必写）收敛到很薄的一层**——T+1 精确解锁、分板涨跌停/手数、过户费、以及"现金分红真账本"这个口径决策。

| 子域 | 自研薄核心 | Qlib + 薄规则层 | 胜方 |
|---|---|---|---|
| 防未来/复权口径/撮合时点 | 自己保证，已出 2 个 bug | 框架保证、实战过 | **Qlib** |
| 公司行为/分红入账 | 全要自己写（现在直接跳过） | 复权口径自带；真现金账本仍需薄补 | **Qlib** |
| 订单回放 | 天然契合 | `order_file`+executor 也支持 | 平 |
| A 股规则（T+1/分板手数/涨跌停） | 自写（已有） | 同样要写，但是在受测内核上做配置/扩展 | 平→偏 Qlib |
| 指标/日历 | 应换成熟库 | Qlib/empyrical | 平 |
| 依赖轻重 | 最轻 | 较重、有学习曲线 | 自研 |
| 两仓统一、少排查 | 否 | **是**（你的目标） | **Qlib** |
| 维护 | 自己背 | 微软+社区 | **Qlib** |

**总体推荐：Qlib 作引擎 + 一层薄的 A 股规则/配置层（承载 T+1 精确语义、分板手数与涨跌停差异、过户费、券商分红账本口径）。** 这层"薄规则"比今天手写的整台引擎小得多，而且是建在受测内核之上的配置/扩展，不再是从零撮合。

诚实的代价：Qlib 较重、有学习曲线；"订单回放"用的是 Qlib 的次要路径（order-execution / 高频执行模块），不如它的"信号→组合"主线成熟好查。这正是要 spike 的原因。

## 4. 唯一真正的不确定点 → 1 天 Spike（先验证再锁定）

锁定 Qlib 前，只需验证一件事：**Qlib 的 order-execution 路径能否干净地承载"外部订单逐分钟回放 + 我们的 A 股约束"，而不被它的信号/组合层束缚。** 具体要回答：

Spike 设计（≈1 天）：取 2 个标的——一个**主板**（窗口内含**除权除息日**）、一个**科创板**——构造一批外部分钟订单（含一次**涨停日买入**、一次**当日买入又当日卖出**的 T+1 违规尝试、一次不足整手），用 Qlib 跑通并核对：

**Pass/Fail 清单**
- [ ] T+1：当日买入当日卖出被拒（或确认需在规则层补，且接得上 Qlib 仓位的"可卖量"）。
- [ ] 涨停买入 / 跌停卖出被拒（以数据涨跌停为准）。
- [ ] 手数：主板 100 倍数、科创 200 起；非法手数被拒或被规整。
- [ ] 复权后 NAV 在除权除息日不出现假跳空；同一 bar 不随窗口漂移（正是现有 C1/C3 要消灭的）。
- [ ] 现金分红口径明确：确认走"复权总收益"还是"现金入账"，并能说清 NAV 语义。
- [ ] 性能：单标的 1 年分钟 + 多标的，内存/耗时可接受。

**判定**：清单基本通过且扩展点顺手 → **锁定 Qlib，两仓统一，ADR-1 改 Accepted**。若 Qlib 在订单回放/ A 股约束上明显较劲 → **回退到薄自研核心**——而它要实现的东西，本文件 §2 已经是现成的规格说明书，`design/01` 的 bug 清单直接变成测试用例。

> 也就是说：无论 spike 结果如何，这份能力图都不浪费——通过则是 Qlib 集成的验收清单，不通过则是自研引擎的实现规格 + 测试集。

## 5. 对两个仓库的影响（含你关于"直接读盘"的意见）

- **统一到 Qlib**：`vortex_data` 直接产出 Qlib 可读的数据落盘（完成它 `design/04` 已规划的 `qlib_view` 导出），`vortex_backtest` 用 Qlib 的 `FileStorage` provider **直接读硬盘数据**。
- **数据访问按你的意见简化**：**不做查询服务跳转**——引擎进程直接读盘（Qlib provider 本就是读本地文件）。这同时取代了原 ADR-2 里"消费 vortex_data HTTP 查询"的方案；原 ADR-2 的价值只剩"别全量读盘"，而 Qlib provider 天然按需读，不再有那个性能问题。
- **边界不变**：策略仍在客户端脚本，服务端只做订单回放/账户模拟；`vortex_data` 仍只管数据、不碰策略。

## 6. 对 ADR-1 的更新与待办

- 本文件**更新了 `design/02` ADR-1 的推荐**：由"方案 C（薄自研核心）"改为"**Qlib + 薄规则层**（待 §4 spike 确认后定为 Accepted）"。ADR-2（数据访问）据 §5 简化为"Qlib provider 直接读盘、无查询服务"。
- 待办：
  1. [ ] 跑 §4 的 1 天 spike，按 Pass/Fail 清单判定。
  2. [ ] spike 通过 → 改 ADR-1 为 Accepted；规划 `vortex_data` 的 Qlib 数据落盘 + `vortex_backtest` 接 Qlib 的薄规则层。
  3. [ ] spike 不通过 → 按本文件 §2 规格走薄自研核心，`design/01` bug 清单转测试集。
  4. [ ] 两种结果都先做 `design/03` 阶段 0（基线提交）与阶段 1（修 C1/C3 口径）——这与引擎选型无关，越早越好。

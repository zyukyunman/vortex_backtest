vortex_backtest 技术文档

独立 HTTP 回测 / 账户回放服务 —— 接收账户、订单、策略配置,基于本地 Tushare 分钟数据做 A 股账户回放,输出成交、拒单、持仓、日净值与汇总报告。本页覆盖系统设计、HTTP 接口协议、环境部署三部分。

1系统设计

vortex_backtest 是一个独立的 HTTP 服务:外部把账户、历史委托(订单)与策略配置提交进来,服务按真实 A 股规则在分钟行情上回放这些委托,产出可对账的成交 / 拒单 / 持仓 / 日净值 / 汇总报告。它只读消费 vortex_data 导出的本地行情,自身不抓数据;第一阶段只做 A 股现金账户、1min 分钟、前复权 qfq 单一口径。

📌

定位是账户回放(replay),不是因子/信号研究:输入是具体委托(symbol/side/qty/日期),不是从数据现算的信号。这条边界决定了引擎选型(见 1.3)。

1.1 整体架构

四层,自上而下:

1.2 引擎选型:自研 replay + 直读 parquet(已去 Qlib)

引擎为自研 A 股分钟撮合引擎,枚举名 replay(前身误名 backtrader,2026-06-07 正名)。Qlib 已从回放关键路径删除(ADR-1 rev.2,2026-06-07 Accepted),原因:

  • 实现现实:所谓"纯 qlib"引擎其实只用了 D.features 读数,撮合 / 账本从未走 Qlib 的 Exchange/Position —— 两次实现都自然地自研了撮合(Qlib 对现金不足是裁剪成部分成交,而本服务的产品语义是拒单)。
  • 运维现实:Qlib 需要 .bin 导出(519MB、6.6 万小文件),pivot 进内存是小服务器扛不住的根因;pyqlib 在 macOS arm64 装不上,被迫维护 amd64-only 容器。
  • 能力对照:T+1 锁仓、分板手数、印花税/过户费分项、拒单 reason code —— Qlib 都不做,无论选型都得自写,而本服务已写好并测试。

Qlib 不报废,降级为"将来真做因子/信号研究时才引入的独立研究线"(NestedExecutor/impact_cost 那时才值钱)。正确性验证改用更权威口径:手算金标准用例(覆盖 A 股规则)+ 真实券商对账单回放。

1.3 数据流与数据集

数据单向流动:vortex_data 抓取/导出 → workspace(parquet)→ vortex_backtest 只读消费。服务读 $VORTEX_DATA_WORKSPACE/data 下的数据集:

数据集用途缺失

stk_limit涨跌停价market_rules_data_missing
suspend_d / stock_st停复牌 / 历史 ST缺表按"无停牌 / 非 ST"处理
instruments / calendar标的主数据 / 交易日历缺表退回代码规则 / 行情日期

全链路 qfq 前复权:指标、买卖、成交、限价、持仓估值、净值都用前复权价;挂单合法性判 raw 价。第一阶段不建模现金分红。

1.4 资源模型

资源说明标识 / 生命周期

1.5 回测作业生命周期

回测异步:提交即入队,后台 worker 执行,客户端轮询到终态再取报告。

queued→ running→ completed| failed| cancelled| interrupted
status含义终态可取报告
queued已入队待执行否否
runningworker 执行中否否
completed成功完成是是
failed失败(带安全错误码)是否
cancelled排队中被取消是否
interrupted重启时残留、已重排恢复是否

取消(POST /backtests/{job_id}/cancel)仅能取消 queued;运行中无法安全中断 → 409。

1.6 A 股撮合口径

  • 成交方式:订单为日级,按 price_type 在当日首/末分钟成交(非日内逐 tick 触价)。
  • T+1:当日买入次日才可卖,违者拒为 t_plus_1_not_sellable。
  • 涨跌停:以 stk_limit 为准,涨停拦买、跌停拦卖。
  • 分板手数:主板 100 股/手,科创 ≥200,北交所另算;不足整手向下取整或拒单。
  • 量能上限:成交量受当日量限制(max_volume_participation)→ 部分成交。
  • 费用:佣金 0.03%(最低 5 元)、印花税 0.05%(仅卖出)、过户费 0.001%、可配 slippage_bps 滑点;均可在 execution 按回测覆盖。
  • 净值起点:所有净值曲线从 1.0 起。

1.7 多策略与报告口径

多策略:一次回测可含多个策略,每个策略是独立子账户(各自初始资金 / 持仓 / 净值),用 params.order_batch_id 绑定各自订单批次,最终汇总聚合展示。

报告口径 = 日级:每日净值 + 当日成交/拒单/持仓 + 摘要指标(收益、最大回撤、已实现盈亏等)。已移除分钟级输出(无 /minutes)。成交记录含 realized_pnl(已实现盈亏)与 requested_quantity(原始下单量,识别部分成交)。

2HTTP 接口协议

对外是 REST/JSON over HTTP,这是唯一的业务接入方式 —— 建账户、下单、提交回测、轮询、取报告全部走 HTTP,命令行只剩 serve(起服务)。Base URL:容器内 http://0.0.0.0:8767,本机默认 http://127.0.0.1:8767。在线可交互:/docs(Swagger)、/redoc、/openapi.json。

2.1 设计原则

  • 纯 HTTP,无 CLI 协议客户端 —— 业务全走接口,易于被 k8s/其它服务编排。
  • 回测异步 —— POST /backtests 入队即回 202 + job_id,轮询到终态再取报告。
  • 只读上游 —— 行情来自 vortex_data 导出的 workspace,回测进程只读。
  • 写接口 fail-closed —— 见 2.6 鉴权。

2.2 开闭环调用时序

用户视角的「开始 → 提交买卖 → 结束 → 关闭 → 报告」全流程,对应示例脚本 scripts/backtest_roundtrip.sh:

建账户

POST /accounts → 201 {account_id, …}

提交买卖(可多次)

POST /accounts/{id}/orders → 201

结束 = 提交回测

POST /backtests → 202 {job_id, status:"queued"}

关闭 = 轮询到终态

GET /backtests/{job_id} 反复轮询

输出报告

GET /backtests/{job_id}/summary | /daily | /trades | /rejections
bash
B=http://127.0.0.1:8767

# ① 建账户
curl -s -XPOST $B/accounts -H 'Content-Type: application/json' \
  -d '{"account_id":"demo","initial_cash":1000000,"name":"演示"}'

# ② 提交买卖(side: 1=买 2=卖)
curl -s -XPOST $B/accounts/demo/orders -H 'Content-Type: application/json' \
  -d '{"order_batch_id":"b1","request_id":"buy-1","trade_date":"2026-05-06","symbol":"600000.SH","side":1,"quantity":1000}'

# ③ 结束回测 = 提交作业(异步,拿 job_id)
JOB=$(curl -s -XPOST $B/backtests -H 'Content-Type: application/json' \
  -d '{"account_id":"demo","order_batch_id":"b1","start_date":"2026-05-06","end_date":"2026-06-05"}' \
  | python3 -c 'import sys,json;print(json.load(sys.stdin)["job_id"])')

# ④ 关闭 = 轮询到终态
until curl -s $B/backtests/$JOB | python3 -c 'import sys,json;s=json.load(sys.stdin)["status"];print(s);exit(0 if s in {"completed","failed","cancelled","interrupted"} else 1)'; do sleep 1; done

# ⑤ 输出报告
curl -s $B/backtests/$JOB/summary
⚡

这一整套已封装进 scripts/backtest_roundtrip.sh(仅依赖 curl + python3):scripts/backtest_roundtrip.sh --symbol 600000.SH --start 2026-05-06 --end 2026-06-05。

2.5 错误码与拒单枚举

HTTP 错误码:400 参数不支持 · 401 未授权 · 403 写接口被禁(非回环且无 token) · 404 资源不存在 · 409 冲突(重复/不可取消) · 422 字段校验失败。

作业安全错误码(failed 时回传,完整堆栈只进服务端日志):

minute_data_missing · adjustment_data_missing · market_rules_data_missing · no_symbols · start_end_required · unsupported_frequency · unsupported_price_adjustment · unsupported_strategy_type · missing_request_payload;其余脱敏为 internal_error。

拒单原因枚举(rejections[].reason,看板做中文化):

suspended · zero_volume · invalid_price_tick · invalid_lot_size · limit_up_buy_blocked · limit_down_sell_blocked · insufficient_cash · insufficient_position · t_plus_1_not_sellable · volume_cap_below_lot。

2.6 写接口鉴权

写接口 = 建账户 / 下单 / 提交回测 / 取消 / 设 meta。规则:

  • 配了 VORTEX_BACKTEST_TOKEN:请求须带 Authorization: Bearer <token> 或 X-Auth-Token: <token>,否则 401。
  • 没配 token:仅本机回环放行;绑到非回环 host(如 0.0.0.0)时写接口直接 403(fail-closed,避免裸暴露)。

只读 / 报告接口不鉴权。对外暴露务必先配 token,再放开绑定地址。

3环境部署

两种跑法:本地 venv(开发 / 跑测试)与 Docker 镜像(部署 / 起服务)。镜像里只启动服务。只想用镜像起服务的话直接看 3.3。规范端口 8767(避开 vortex_data 的 8765)。

⚠️

代码用了 3.11+ 的 enum.StrEnum —— 必须 Python 3.12 / 3.13;系统自带 3.9/3.10 会报 cannot import name 'StrEnum'。

3.1 本地 venv(开发 / 测试)

bash
cd /path/to/vortex_backtest
python3.13 -m venv .venv            # 或 python3.12
.venv/bin/python -m pip install -U pip
.venv/bin/python -m pip install -e '.[dev]'

起好后:看板 /ui/ · 交互式 API /docs · 本文档 /guide。回测调用见 2.2(脚本或 curl)。

3.2 Docker / compose(部署,推荐)

应用镜像 FROM vortex-base(统一依赖底座,由 vortex_common 构建),只叠本仓代码(pip install --no-deps),改代码秒级重建。前提:基础镜像 vortex-base:latest 必须已存在:

bash
(cd ../vortex_common && scripts/build-base-image.sh)   # 仅首次 / base 变更时

cd /path/to/vortex_backtest
cp .env.example .env                # 改 VORTEX_BACKTEST_WORKSPACE 指向你的数据目录
docker compose up -d --build        # 构建本仓代码并起服务(端口默认 8767)
curl http://127.0.0.1:8767/health   # {"status":"ok"}
docker compose logs -f              # 看日志
docker compose down                 # 停服务
  • 卷:/workspace = 数据(默认只读挂载,来自 VORTEX_BACKTEST_WORKSPACE);/state = 账户 / 订单 / 作业 / 报告(可写)。
  • 端口:默认只绑 127.0.0.1:8767(仅本机)。要对外暴露:.env 设 VORTEX_BACKTEST_BIND_ADDR=0.0.0.0,并先配 VORTEX_BACKTEST_TOKEN(写接口鉴权)。

不用 compose 的等价命令:

bash
docker build --build-arg BASE_IMAGE=vortex-base:latest -t vortex-backtest:latest .
docker run -d --name vortex-backtest \
  -p 127.0.0.1:8767:8767 \
  -v /path/to/workspace:/workspace:ro \
  -v "$(pwd)/state:/state" \
  vortex-backtest:latest

3.3 组合镜像与 k8s(一进程一容器)

整个 Vortex 的发布形态(vortex_common ADR-001):一个版本化镜像 vortex:<tag> 内含全部服务代码,运行时每服务一个容器(与 k8s 对齐,崩溃隔离、独立重启 —— qmt 实盘与 backtest 重算彻底分开)。

bash
# 在 vortex_common:按 deploy/versions.yml 各仓 tag 造组合镜像
deploy/build-release.sh
# 每服务一容器(同一镜像 + 不同 command: vortexctl <svc>)
docker compose -f deploy/docker-compose.yml up -d

启动契约:common 的 vortexctl backtest 优先执行本仓 deploy/run.sh(自描述如何起自己),缺失则回退内置默认 vortex-backtest serve。规范端口(容器内):

服务端口命名卷
vortex_data8765vortex-workspace(读写)
vortex_qmt8810vortex-qmt-state
vortex_backtest8767vortex-workspace(只读)+ vortex-backtest-state
vortex_trader(预留)8820vortex-trader-state

升级 k8s:每个 compose 服务 → 一个 Deployment(同镜像换 command)。镜像内非 root 用户 vortex(uid 1000)+ 命名卷规避 bind-mount uid 错配。

3.5 迁移到云服务器

  1. 服务器装 Docker Engine + compose 插件(docker info 正常)。
  2. 拉代码(或把镜像推私有 registry 再拉);先在该机构建 vortex-base。
  3. 准备数据目录(来自 vortex_data 导出),把 .env 的 VORTEX_BACKTEST_WORKSPACE 指过去。
  4. docker compose up -d --build → curl http://127.0.0.1:8767/health。
  5. 对外服务:配 VORTEX_BACKTEST_TOKEN + VORTEX_BACKTEST_BIND_ADDR=0.0.0.0,前置反代/防火墙。
